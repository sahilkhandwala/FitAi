🥗 Nutrition Tracker — Design Spec

Date: 2026-05-25  ·  Replaces: jolly-whistling-nebula.md

⭐ North Star

Help Sahil reduce A1C, lower LDL cholesterol, increase HDL, and stay physically fit.

  • Make it effortless to log what Sahil eats — no gaps in the data
  • Surface patterns and correlations between food choices and health markers
  • Deliver personalised, actionable nudges grounded in actual logged history
  • Keep Sahil on track toward his long-term health targets, not just report on the past

Every message, every nudge, every report should connect back to this north star. The bot is not a meal logger — it is an active health accountability partner.

Architecture Overview

Two-process Python application on a single GCP VM, sharing a local SQLite database.

Process 1: Bot (python-telegram-bot v20+)
  └── Handles all real-time Telegram interactions

Process 2: Prefect Flows (scheduled jobs)
  └── 5 Prefect flows with scheduling, retries, and observability

Shared:
  ├── SQLite (WAL mode) — meal data, health profile, reports, LangGraph state
  ├── knowledge_base/   — JSON files, one per ingested research article
  ├── cachetools TTLCache — media group buffering (2s), health profile (1hr)
  └── diskcache          — callout dedup (24hr), Fitbit token rotation

Context Injection vs Tool-call

Two ways an agent accesses shared data — the right choice depends on whether the context is always consumed or only sometimes needed:

PatternHowWhen to use
Injection (context: in YAML)Loaded from DB once, prepended to system prompt before agent runsContext is always consumed — cheaper than a tool-call round-trip
Tool-call (tools: in YAML)Agent calls the tool mid-run, on demandContext is conditionally needed — avoid paying tokens when routing or task doesn't require it

HealthExtractorAgentNonePure document extraction — profile doesn't help Claude read a blood test
KnowledgeIngestorAgentNonePure ingestion — profile doesn't change how a research paper is summarised
PatternDetectorAgentNoneStreak thresholds are hardcoded, not personalised to lab values

Web search policy: Never fires automatically. Any agent must first send a permission request via Telegram and wait for explicit confirmation before calling web_search.

Pause-and-wait — LangGraph interrupt(): Both ask_web_search_permission and confirm_with_user use LangGraph's built-in interrupt/resume. AsyncSqliteSaver holds graph state during the pause. When the user replies, the bot handler calls graph.invoke(Command(resume=user_message), config={"thread_id": chat_id}) to resume. No custom state machine needed.

def ask_web_search_permission(reason: str) -> str:
    answer = interrupt(f"I'd like to search the web for {reason}. Can I?")
    return answer  # resumes here when user replies

def confirm_with_user(message: str) -> str:
    answer = interrupt(message)
    return answer

Feature 1 — Meal Photo Logging

  • Telegram album photos arrive separately with a shared media_group_id
  • First photo schedules call_later(2, process_group, id) — fires after 2s, reads all buffered photos as one meal
  • Single meal-type keyboard shown: [Breakfast] [Lunch] [Dinner] [Snack] — never inferred
  • Non-food detected → "That doesn't look like food — did you mean to send this?" with [Yes, analyze anyway] [Discard]
  • Beverages included — sugar/alcohol flagged for A1C relevance
  • No image storage — analyzed on receipt, discarded immediately

Analysis stored in SQLite (not sent to user):

{
  "foods_identified": ["white rice", "chicken curry", "chai"],
  "estimated_macros": { "calories": 780, "protein_g": 38, "carbs_g": 95,

Feature 2 — Health Profile Upload

  1. User sends a lab report PDF
  2. HealthExtractorAgent extracts: A1C, LDL, HDL, triglycerides, medications, targets, BMI
  3. Bot confirms: "I found: A1C: 6.2%, LDL: 142 mg/dL, HDL: 48 mg/dL. Does this look right? [Confirm] [Re-upload]"
  4. On confirm: INSERT INTO user_health_profile (report_date, a1c, ...) — one row per lab visit, history preserved
  5. On confirm: user_nutrition_guidance fully replaced with rules derived from the new lab values

Null state (day 1 before any PDF): agents inject fallback context: "No lab report uploaded yet — recommendations are general." One-time prompt fires on first non-photo, non-PDF message; tracked via diskcache key onboarding_prompt_sent.

Feature 3 — Morning Briefing (10:30am PST)

Simple Prefect flow — one Claude call, no LangGraph agent.

  • Steps yesterday (Fitbit /activities/date/{yesterday})
  • Sleep last night (Fitbit /sleep/date/{today})
  • Weather: Open-Meteo, lat 37.3861, lon -122.0839 — rain window included only if rain expected
  • Quote (alternates by day): even days → Bhagavad Gita verse + brief context; odd days → tech/business leader quote
  • Steps nudge if previous day < 5,000 steps
  • Sleep nudge if last night < 7 hours

Feature 5 — Daily Meal Summary (11:30pm PST)

  • All meals logged today with foods identified
  • Total daily macros (calories, protein, carbs, fat, fiber, saturated fat, sugar)
  • A1C risk foods + alternatives  ·  Cholesterol risk items flagged
  • Daily dietary score (0–100)
  • One concrete improvement for tomorrow
  • Steps and sleep not included

Feature 6 — Pattern Detection

Runs inside daily_summary_flow after each summary. Sunday weekly_report_flow also triggers it for 7-day patterns.

PatternThreshold
High glycemic load≥3 consecutive days → escalating callout
Saturated fat above target≥3 consecutive days → escalating callout
Skipped meals≥3 consecutive days → escalating callout
Steps below 5,000≥3 consecutive days → escalating callout
Sleep under 6hrs≥3 consecutive days → escalating callout

Dedup: diskcache key callout:{date}:{pattern_type} 24hr TTL. Delivery: random offset 8–11am next morning.

Escalation — three tiers:

StreakToneWhat changes

Days 4–6FirmStronger language, explicit impact on A1C/cholesterol goal

Feature 7 — General Conversation

Handles all text not classified as a health symptom, command, or known trigger. Answers personalised to actual health values. Out-of-scope questions get a warm redirect.

Classification rule: Personal symptoms ("why am I tired?") → HealthInsightsAgent. General questions ("is oat milk good for cholesterol?") → answered directly by OrchestratorAgent using health profile and knowledge base context.

Feature 8 — Health Insights Q&A

Triggered by natural language symptom questions. Lookback: always 2 days.

  1. Queries last 2 days of meal logs, steps, sleep from SQLite
  2. Asks one focused follow-up question
  3. Incorporates user answer + data → personalised explanation
  4. Asks web search permission before validating correlations
  5. Delivers evidence-based answer citing specific meals/metrics

Conversation state stored in LangGraph SQLite checkpoint (AsyncSqliteSaver).

Feature 9 — Weekly Report (Sunday 8pm PST)

  • Week-over-week dietary score delta
  • Worst patterns of the week  ·  Avg daily steps + trend  ·  Avg sleep + trend
  • Recommendation follow-through: last week's stored recommendations vs this week's meals → followed / partially / not followed
  • Correlations (e.g. worst dietary days ↔ lowest step counts)
  • Honest verdict: one warm direct sentence about week's trajectory toward A1C/cholesterol goals
  • 3 specific ranked changes for next week

Week 1 exception: skip follow-through section. User can send "skip comparison" before Sunday to flag omission.

Feature 10 — Episodic → Semantic Memory Extraction (Sunday 6pm PST)

Runs via semantic_extraction_flow at 6pm Sunday — 2 hours before the weekly report — so WeeklyReportAgent benefits from freshly extracted semantic facts.

  • Queries last 90 days of meal_logs, daily_health_logs, and patterns
  • Claude extracts behavioural facts grouped by category: meal_pattern, sleep_pattern, activity_pattern, behavioral
  • Full replacement: old facts replaced with updated ones in user_semantic_memory
  • Meaningful after ~4 weeks of data; before that, agents fall back to raw recent data only

Agents that use semantic memory: WeeklyReportAgent, HealthInsightsAgent, OrchestratorAgent, MealAnalyzerAgent

Feature 11 — User Onboarding

Triggered on first message when user_profile row doesn't exist (checked via diskcache key onboarding_complete). Bot sends a short conversational setup before processing any other request:

Bot: Hey Sahil! 👋 Let me personalise your experience with a few quick questions.

     What's your diet type?
     [🥩 Omnivore]  [🥦 Vegetarian]  [🌱 Vegan]

Bot: Any food allergies or things you strictly avoid?
     (Type them comma-separated, or tap 'None')  [None]

Bot: What's your typical activity level?
     [🪑 Sedentary]  [🚶 Moderate]  [🏃 Active]

Bot: Last one — what's your daily calorie target?
     [1,600 kcal]  [1,800 kcal]  [2,000 kcal]  [Skip for now]

What gets populated: diet_type, allergies, avoided_foods, activity_level, calorie_target. Other fields (height_cm, weight_kg, macro breakdowns) default to NULL and are refined via /profile update.

Skip / resume: If user sends a non-answer mid-onboarding (e.g. sends a photo), save partial answers and resume after that interaction completes.

Knowledge Base

Two sources ground agent recommendations in facts rather than LLM guesses:

Indian Food Nutrition Table

Structured SQL lookup. MealAnalyzerAgent queries by food name for accurate macros on Indian dishes Claude has limited training data for. Populated from IFCT / NIN India before deployment.

Research Articles — Structured Extraction

Peer-reviewed PDFs on A1C, LDL, HDL, dietary interventions. At ingest: KnowledgeIngestorAgent uses Claude to extract 10–20 key findings and relevance tags, writing one JSON file to knowledge_base/. At query time: all files are loaded (~10–20 docs, ~6,000 tokens) and injected directly into the agent's context — no retrieval step, full coverage guaranteed.

Message Tone & Style

  • Include relevant emojis throughout — never feel dry or robotic
  • Warm, conversational — like a knowledgeable friend, not a clinical report
  • Concise but human — avoid bullet-point walls of text
  • Vary phrasing across days so messages don't feel repetitive
✅ Good
"Great job logging all three meals today! 🎉 Your dietary score is 72/100 — solid, but that naan at lunch was doing your A1C no favours 😅. Tomorrow: try swapping it for a small roti."
❌ Bad
"Daily Score: 72. Flags: refined carbs. Recommendation: reduce naan."
